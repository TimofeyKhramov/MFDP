# Бизнес-анализ продукта и работа с требованиями

Пока не внёс замечания с консультации

## Пункт 1 «Основная гипотеза»

- Продукт (внутренний): сервис для анализа проектной документации
- Сегмент: строительная компания, занимающаяся витражными конструкциями
- Сценарий: в компании анализ ПД проводится вручную. То есть приходит ПД на объект, далее сотрудник осуществляет поиск чертежей в данной ПД и вручную переносит необходимую информацию: размеры, наименование.
- Метрика: производительность (увеличивается до 2 минуты/чертёж, было 6 минут/чертёж)
- Причина: автоматизация анализа ПД

Для инженера-проектировщика фасадной компании, работающего с ПД на витражные конструкции, использование сервиса автоматического извлечения позиций и габаритных размеров из PDF-файлов позволит сократить время обработки одного чертежа с 6 до 2 минут (рост производительности в 3 раза в течение часа работы) за счёт исключения ручного поиска, выписывания и сверки данных.

## Пункт №2 «Метрики»

Продуктовые метрики

| Метрика | Способ измерения | baseline | Частота |
|---------|----------------|----------|----------|
| Общее время анализа одного чертежа | Логирование времени, модуль time | Ручной замер |1 раз/чертеж|
| Общее время анализа одного экземпляра ПД | Логирование времени, модуль time | Ручной замер |1 раз/экземпляр|
| Количество исправленных полей на одном чертеже | Регистрация сервисом | - |1 раз/чертеж|
| Доля принятых данных | | |1 раз/чертеж|

Технические метрики

| Метрика | Способ измерения | baseline | Частота |
|---------|----------------|----------|----------|
| accuracy | Регистрация сервисом, scikit-learn | - |1 раз/чертеж|
| precision | Регистрация сервисом, scikit-learn | -|1 раз/чертеж|
| recall |Регистрация сервисом, scikit-learn | - |1 раз/чертеж|

## Пункт №3 «Элементы прототипа»

1)	Войдёт в прототип
   
- Графический интерфейс с кнопкой «Загрузить PDF-файл»
- Поддержка загрузки только одного файла
- Функция ввода страниц для анализа (то есть содержащих чертежи)
- Анализ только основных параметров: габаритные размеры, наименование

2)	Не войдёт в прототип:
- Загрузка нескольких файлов для анализа
- Автоматическое определение страниц для анализа
-	Извлечение материала конструкций
  
Причина: возможность проверки основной гипотезы без данных пунктов. Данные пункты могут повысить производительность, однако основная потребность заказчика заключается именно в анализе наименования и габаритных размеров. 

## Пункт №4 «Блоки»

1)	Готовые блоки
- OCR-системы
- модули для считывания текста из pdf (pdf-plumber)
- LLM/VLLM-модели

2)	Самостоятельная реализация
- подготовка датасета (отсутствуют необходимые чертежи в открытом доступе как единый датасет)
- разметка данных (для извлечения параметров с нужных позиций), дообучение LLM/VLLM

## Пункт №5 «Риски»

1)	Низкое качество распознавания: значения по одной из метрик (accuracy, recall, precision) < 0.9
2)	Долгое время анализа одного чертежа (> 4 минут), что может быть связано с ограничениями по ресурсам железа
3)	Маленький исходный набор данных

## Пункт №6 «Конечный вид прототипа»

- Пользовательское окно с кнопкой загрузки файла и вводом страниц для анализа
- Встроенная модель (vllm, дообучение) для анализа чертежа
- Вывод итоговых значений в формате CSV (генерация файла csv, возможно, прямая запись в БД)
- Упаковка сервиса в Docker

## Пункт №7 «Критерии остановки теста прототипа»
- анализ одного чертежа > 4 минут
- значения метрик < 0,9
